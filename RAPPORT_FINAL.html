1. Introduction

Dans le cadre de notre projet d'étude au sein du groupe MBN Global Education, nous avons entrepris la conception et le développement d'une plateforme web complète destinée aux étudiants, enseignants et administrateurs de l'ensemble des campus du groupe MBN.

Ce projet s'inscrit dans une démarche de modernisation des outils de communication internes à l'établissement. L'idée est de proposer un réseau social éducatif qui centralise les échanges, les événements, les clubs et l'organisation du travail personnel au sein d'une seule application.

Ce rapport final présente l'état d'avancement complet du projet, les choix technologiques adoptés, l'architecture technique détaillée, les fonctionnalités développées ainsi que la stratégie de déploiement mise en place pour rendre l'application accessible à tous les utilisateurs du groupe MBN.

Note : Ce rapport fait suite au Rapport N°1 qui présentait les maquettes initiales et le prototype en HTML/CSS/JavaScript. Depuis, le projet a été entièrement repensé et reconstruit avec des technologies professionnelles (React + Django).

4. Architecture Technique

4.1 Architecture générale

L'application suit une architecture client-serveur séparée en deux parties indépendantes qui communiquent via une API REST en échangeant des données au format JSON.

┌─────────────────────┐ ┌─────────────────────┐ │ FRONTEND │ HTTP │ BACKEND │ │ React + Vite │ <-------> │ Django + DRF │ │ Port 5173 │ /api │ Port 8000 │ │ │ │ │ │ - Pages React │ │ - API REST (30+ │ │ - Composants │ │ endpoints) │ │ - CSS Design System │ │ - Auth JWT │ │ - React Router │ │ - ORM Django │ └─────────────────────┘ │ │ │ │ ┌──────┬───────┐ │ │ │ SQLite / │ │ │ │ PostgreSQL │ │ │ └─────────────┘ │ └─────────────────────┘

Principe : Le navigateur de l'utilisateur exécute le code React (frontend). Lorsqu'il a besoin de données (connexion, messages, clubs, etc.), React envoie une requête HTTP au backend Django. Celui-ci traite la demande, interagit avec la base de données, et renvoie une réponse au format JSON.

5. Backend — Django REST Framework

5.1 Pourquoi Django ?

Django est un framework web Python créé en 2005, utilisé par des entreprises comme Instagram, Spotify et Mozilla. Nous l'avons choisi pour les raisons suivantes :

  • ORM intégré : permet de manipuler la base de données avec du code Python au lieu d'écrire du SQL brut
  • Système de migrations : gère automatiquement l'évolution de la structure de la base de données
  • Sécurité intégrée : protection contre les injections SQL, les attaques CSRF et le XSS
  • Interface d'administration : panneau d'administration automatique pour gérer les données
  • Django REST Framework (DRF) : extension qui transforme Django en API REST complète

Django REST Framework ajoute à Django la capacité de créer des API. Il fournit des Serializers (convertisseurs Python ↔ JSON), des Views (logique de traitement), et un système de Permissions (contrôle d'accès).

5.2 API REST — Endpoints

5.3 Authentification JWT (JSON Web Token)

L'authentification est gérée par le système JWT (JSON Web Token). C'est un standard ouvert (RFC 7519) utilisé par la majorité des applications web modernes.

Fonctionnement

1. L'utilisateur envoie email + mot de passe | v 2. Le serveur vérifie les identifiants dans la base de données | v 3. Si correct, le serveur génère 2 tokens : +----------------------------------------------+ | ACCESS TOKEN (valide 7 jours) | | Envoyé à chaque requête dans le header HTTP | | "Authorization: Bearer eyJhbGciOi..." | +----------------------------------------------+ | REFRESH TOKEN (valide 30 jours) | | Permet de renouveler l'access token expiré | +----------------------------------------------+ | v 4. Le frontend stocke ces tokens dans localStorage | v 5. À chaque requête, Axios ajoute automatiquement le token dans les headers HTTP | v 6. Le backend vérifie le token et identifie l'utilisateur

Pourquoi JWT et pas les sessions classiques ?

  • Stateless : le serveur ne stocke rien en mémoire. Toutes les informations sont dans le token lui-même.
  • Adapté aux API REST : pas besoin de cookies, ce qui fonctionne mieux avec les applications mobiles et les SPA (Single Page Application).
  • Sécurisé : le token est signé numériquement avec une clé secrète. Toute modification le rend invalide.
  • Scalable : plusieurs serveurs peuvent vérifier le même token sans partager de session.

Exemple concret : Connexion

// Requête envoyée par le frontend POST /api/auth/login/ Body : { "email": "mboussi@mbn.edu", "password": "password123" } // Réponse du backend { "success": true, "token": "eyJhbGciOiJIUzI1NiIs...", "user": { "id": 1, "nom": "MBOUSSI", "prenom": "Marc", "ecole": "ESIIA", "role": "directeur" } }

6.2 Relations entre les tables

USER ──1────N──> MESSAGE (un utilisateur envoie plusieurs messages) USER ──1────N──> TODO (un utilisateur a plusieurs tâches) USER ──N────N──> CLUB (via CLUBMEMBER : many-to-many) USER ──N────N──> EVENT (via EVENTPARTICIPANT : many-to-many) USER ──1────N──> CLUB (created_by : un utilisateur crée plusieurs clubs) USER ──1────N──> EVENT (created_by : un utilisateur crée plusieurs événements)
Clé étrangère (FK) : une référence vers une autre table. Par exemple, sender = FK → User signifie que chaque message est lié à un utilisateur expéditeur.
ManyToMany : une relation où un enregistrement peut être lié à plusieurs enregistrements de l'autre table (et vice-versa). Exemple : un utilisateur peut être membre de plusieurs clubs, et un club peut avoir plusieurs membres.

7. Frontend — React

7.1 Pourquoi React ?

React est une bibliothèque JavaScript créée par Facebook (Meta) en 2013, utilisée par Facebook, Instagram, Netflix, Airbnb et des milliers d'autres entreprises. Nous l'avons choisi pour les raisons suivantes :

  • Composants réutilisables : l'interface est découpée en briques indépendantes (Navbar, Login, Messages, etc.)
  • Virtual DOM : React ne modifie que les parties de la page qui changent, ce qui rend l'application très rapide
  • Vaste écosystème : des milliers de bibliothèques compatibles (Router, Axios, etc.)
  • Compétence très demandée sur le marché du travail

Vite est l'outil de build qui compile notre code React. Il offre un démarrage instantané et un rechargement automatique lors du développement (Hot Module Replacement).

10. Sécurité

La sécurité est un aspect fondamental de l'application. Voici les mesures mises en place :

MesureDétail
Mots de passe hashés Django utilise l'algorithme PBKDF2 + SHA256 avec salage aléatoire. Les mots de passe ne sont jamais stockés en clair dans la base de données. Même si la base est compromise, les mots de passe restent illisibles.
Tokens JWT signés Les tokens sont signés numériquement avec une clé secrète (HMAC-SHA256). Toute tentative de modification rend le token invalide.
CORS (Cross-Origin) Seul le frontend autorisé peut communiquer avec le backend. Les requêtes provenant d'autres sites sont bloquées.
Validation des uploads Les photos de profil sont validées : type MIME (JPEG, PNG, GIF, WebP uniquement) et taille maximale de 5 Mo.
Permissions Un utilisateur ne peut modifier ou supprimer que ses propres données. Les endpoints sont protégés par des vérifications d'identité.
Expiration des tokens L'access token expire après 7 jours, le refresh token après 30 jours. Un utilisateur inactif est automatiquement déconnecté.
Protection CSRF Django protège nativement contre les attaques Cross-Site Request Forgery.
Protection injection SQL L'ORM de Django paramètre automatiquement les requêtes SQL, empêchant toute injection.

Compétences acquises

Ce projet nous a permis de développer des compétences concrètes en :

  • Développement Full-Stack : maîtrise du frontend (React) et du backend (Django)
  • Conception d'API REST : architecture, endpoints, sérialisation JSON
  • Gestion de base de données : modélisation, relations, migrations
  • Sécurité web : authentification, hashage, CORS, validation
  • Déploiement cloud : mise en production sur des services d'hébergement modernes
  • Gestion de projet : planification, itérations, documentation

Perspectives d'amélioration

Pour aller plus loin, le projet pourrait évoluer vers :

  • Notifications push en temps réel (WebSockets)
  • Application mobile native avec React Native
  • Forum de discussion par école ou par formation
  • Partage de supports de cours entre professeurs et étudiants
  • Système de notifications par email
